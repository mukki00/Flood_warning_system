Introduction

The second coursework for "Cloud Computing and Internet of Things" requires a prototype utilising both IoT and Cloud. The system developed is an IoT-Enabled Urban Flood Early Warning and Alert System.

An ESP32 (simulated via Wokwi) collects water level, temperature, humidity, rain intensity, and soil moisture. Data flows to a Flask REST API over HTTP POST and MQTT, is cached in Redis locally, and optionally synced to Azure Cosmos DB and Event Hub for cloud analytics.

An offline-first architecture ensures zero data loss under intermittent cloud connectivity, with a background retry worker that drains the Redis queue when connectivity is restored. A React/Vite dashboard renders live readings and tri-level risk alerts.

Aims & Objectives

Deliver a resilient end-to-end flood early warning platform combining edge sensing, local-first persistence, and cloud analytics:

  • Simulate ESP32 node (Wokwi) with ultrasonic, DHT22, rain and soil sensors.
  • Implement tri-level risk classification LOWMODERATEHIGH on-device.
  • Develop Flask REST API for HTTP POST ingestion and MQTT subscription.
  • Build offline-first store (in-memory + Redis) with cloud retry queue.
  • Integrate Azure Cosmos DB (persistent storage) and Event Hub (streaming).
  • Build React/Vite dashboard with sensor cards, time-filtered charts, alert log.
  • Provide PowerShell operational scripts for Redis startup and offline checks.

System Flow

Software Design & Development

Three software tiers work together: ESP32 firmware, Flask backend, and React frontend.

① ESP32 Firmware (C++ / Arduino)

  • measureWaterLevel() — triggers HC-SR04, converts pulse to cm distance.
  • readDHT() — reads temperature (°C) and humidity (%) from DHT22.
  • readADC(pin) — reads 12-bit value from rain/soil potentiometer pins.
  • computeRisk() — weighted multi-sensor scoring → LOW / MODERATE / HIGH.
  • updateIndicators(risk) — drives LED pins, buzzer, and LCD1602 over I2C.
  • postToFlask(payload) — serialises JSON and HTTP POSTs to Flask API.
  • publishMQTT(payload) — publishes sensor JSON to flood/node1/data.
  • reconnectMQTT() — auto-reconnect loop for HiveMQ broker.

② Flask REST API (Python)

  • ingest_sensor_data() — validates 7-field payload, stores locally, syncs cloud, auto-generates MODERATE/HIGH alerts.
  • get_latest() — returns most recent reading from in-memory store.
  • get_alerts() — returns alert log; supports ?limit= query param.
  • get_status() — returns system health: Redis state, cloud targets, queue depth.
  • start_mqtt_client() — background thread subscribing to HiveMQ broker.
  • sync_reading_to_cloud() — writes to Cosmos DB + Event Hub; returns bool.
  • start_sync_worker() — drains Redis retry queue on cloud reconnect.
  • offline_store.enqueue_sync_reading() — queues failed syncs to Redis.

③ React Dashboard (JavaScript / Vite)

  • useSensorData(filterKey) — polls /api/* every 5 s; manages state for latest, history, alerts, status.
  • SensorCard — colour-coded metric tiles (water level, temp, humidity, rain).
  • RiskIndicator — animated badge; pulses at HIGH risk.
  • SensorChart — Recharts line chart of history with time-window filter.
  • AlertList — scrollable MODERATE/HIGH alert log with timestamps.
  • TimeFilter — Live / 1 h / 6 h / 24 h history views.

④ Azure Cloud Integration

  • azure_cosmos_client.py — upserts to FloodWarningDB/SensorReadings; range queries by timestamp.
  • azure_eventhub_client.py — batches and publishes to flood-events Event Hub.
  • cloud_sync.py — orchestrates dual-target sync; composite success flag.
  • config.py — all tuneable thresholds, Redis keys, Azure endpoints via .env.

⑤ Offline Resilience Scripts

  • start-redis.ps1 — pulls and starts Docker Redis container automatically.
  • offline-check.ps1 — verifies Redis reachability, API health, and cloud sync status.
  • OFFLINE_RUNBOOK.md — step-by-step guide for fully air-gapped local operation.

Conclusion

This prototype successfully delivers a complete IoT-to-Cloud flood early warning pipeline — from simulated ESP32 edge sensing to real-time cloud analytics and a live React monitoring dashboard.

The offline-first architecture (Redis + retry worker) ensures zero data loss under intermittent connectivity — critical for emergency systems. Tri-level risk classification with local LED/buzzer actuation and remote dashboard alerts provides end-to-end flood monitoring applicable to real urban deployments.

The project deepened practical expertise in IoT firmware (C++/Arduino), RESTful API design (Python/Flask), cloud-integrated data pipelines (Azure Cosmos DB + Event Hub), and frontend real-time monitoring (React/Vite) — all core themes of the CS7080 module.
